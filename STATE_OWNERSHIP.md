# State Ownership

This map describes where House state lives today, who should eventually own it,
and the smallest useful migration step. It exists to prevent accidental runtime
state from becoming invisible infrastructure.

## Residents / Personas

- Current source of truth: `src/houseData.ts` seeds, live React runtime, localStorage.
- Storage class: localStorage plus exportable config files in `config/personas/`.
- Eventual owner: Steward for runtime placement/state, UI for edits, persona for
  subjective prompt/identity details, coder only through explicit tools.
- Current risks: edits can diverge between runtime, config export, and seeds.
- Minimal next migration: load `config/personas/*.json` on explicit load and then
  later on startup behind a recovery-safe toggle.

## Rooms / Floors

- Current source of truth: `src/houseData.ts` seeds, live React runtime, and
  Steward-edited room config files.
- Storage class: localStorage plus direct room config files in `config/rooms/`.
- Eventual owner: Steward for room state and occupancy; UI/coder for structure.
- Current risks: blueprint placement is default-only; no visual coordinate
  editor or deletion/retirement flow yet.
- Minimal next migration: add guarded blueprint coordinate editing and explicit
  room retirement.

## Movement / Activity

- Current source of truth: live runtime generated by `useHouseRuntime`.
- Storage class: localStorage and `state/runtime.json` on export.
- Eventual owner: Steward.
- Current risks: high-churn activity is not file-backed as its own ledger.
- Minimal next migration: write Steward activity/House events to append-only log
  separate from runtime snapshots.

## Time Controls

- Current source of truth: live runtime `config`.
- Storage class: localStorage and runtime export.
- Eventual owner: UI-only for controls; Steward reads presence/time state.
- Current risks: time mode and accelerated state can surprise memory generation.
- Minimal next migration: include a clear runtime banner in export/load metadata.

## Persona Memories

- Current source of truth: live runtime plus file-backed archive.
- Storage class: localStorage, per-memory JSON files under
  `state/personas/<persona-id>/memories/day-###/`, aggregate JSONL archive at
  `state/memories/persona-memories.jsonl`, backup/export through `state/`.
- Eventual owner: persona-owned subjective memory, with Steward/Librarian
  indexing and consent enforcement.
- Current risks: older localStorage memories may predate validation/source fields.
- Minimal next migration: validate and merge file-backed archive on startup, then
  eventually prefer file archive over localStorage.

## Unclaimed Imports

- Current source of truth: `personaMemories` entries whose `personaId` has no
  matching resident.
- Storage class: localStorage and file-backed persona memory archive.
- Eventual owner: Steward until claimed; then persona-owned.
- Current risks: imported names can pollute recall if claimed incorrectly.
- Minimal next migration: keep unclaimed imports excluded from Librarian indexing
  until housed/claimed.

## Librarian Records

- Current source of truth: `state/librarian/records.jsonl`.
- Storage class: file-backed append-only JSONL with tombstone/compact.
- Eventual owner: Librarian.
- Current risks: query is token search only; permissioned recall is not complete.
- Minimal next migration: implement visibility resolver before richer recall.

## House Event Log

- Current source of truth: live runtime `houseEvents`, promoted from activity,
  plus file-backed archive.
- Storage class: localStorage, runtime export, and append-only
  `state/house/events.jsonl`.
- Eventual owner: Steward, indexed by Librarian.
- Current risks: event promotion is still heuristic and activity-derived.
- Minimal next migration: explicit event creators for meals, absences, guests,
  failures, and major relationship changes.

## Relationship Updates

- Current source of truth: live runtime `relationshipUpdates`, derived from
  social House events, plus file-backed archive.
- Storage class: localStorage, runtime export, and append-only
  `state/relationships/updates.jsonl`.
- Eventual owner: Steward for detecting social deltas; Librarian for indexing
  and recall; personas for subjective interpretation in memory.
- Current risks: updates are heuristic and symmetrical when derived from shared
  events; they do not yet capture private direct-room shifts.
- Minimal next migration: route visible relationship updates through the
  visibility resolver before adding richer relationship simulation.

## Weather / Outside Signal

- Current source of truth: live runtime `outsideSignals`, Open-Meteo/manual
  refresh.
- Storage class: localStorage and runtime export.
- Eventual owner: Steward as shared external anchor.
- Current risks: default location is hardcoded; signal history is shallow.
- Minimal next migration: persist location/settings separately.

## Conversation / Direct Room State

- Current source of truth: live runtime `conversation`, `directRooms`, and
  `roomConversations`.
- Storage class: localStorage and runtime export.
- Eventual owner: Steward for routing; persona/direct room boundary for private
  transcripts.
- Current risks: direct room transcripts are not separately file-backed.
- Minimal next migration: append direct room turns to per-room or per-persona
  transcript files before model call history grows.

## Settings / Model IDs / API Availability

- Current source of truth: persona model fields, persona `apiEnabled` activation
  flags, runtime config, and Electron main environment for API key availability.
- Storage class: localStorage/config export for model ids and per-resident API
  activation; environment for key.
- Eventual owner: UI-only settings with Steward-readable availability.
- Current risks: placeholder model ids may not match future provider reality;
  per-resident API activation is persona-state backed rather than a dedicated
  provider settings ledger.
- Minimal next migration: central model registry with display labels and provider
  validation, plus file-backed persona config for activation state.

## Backups / Export / Load / Restore

- Current source of truth: Electron main process file operations.
- Storage class: `backups/`, `config/`, `state/`, docs snapshots.
- Eventual owner: Steward/coder infrastructure.
- Current risks: restore reloads runtime but not every future file-backed ledger
  unless included under `state/`.
- Minimal next migration: keep all durable ledgers under `state/` or `config/`
  so backup coverage remains simple.
